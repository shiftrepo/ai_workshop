## 1. テーブル駆動設計とは

### 問題：if-else の組み合わせ爆発

```python
# ❌ BAD: 18個のif-elseが必要
if app == "A" and user == "個人":
    fee = 80000
elif app == "A" and user == "法人":
    fee = 150000
elif app == "A" and user == "行政":
    fee = 60000
# ... × 6種 × 3区分 = 18個のif文
```

### 解法：テーブル駆動

```python
# ✅ GOOD: テーブルに定義、参照は1行
BASE_FEE_TABLE = {
    ("A", "個人"):  80_000,
    ("A", "法人"): 150_000,
    ("A", "行政"):  60_000,
    # ... 18エントリ
}

fee = BASE_FEE_TABLE[(app_type, applicant_type)]
```

### 3原則

| 原則 | 内容 |
|------|------|
| **① 分離（Separation）** | 「何が正しいか（テーブル）」と「どう計算するか（ロジック）」を完全に分離。料金テーブルは経営・法務が管理し、計算ロジックはエンジニアが管理できる |
| **② 網羅（Completeness）** | 全ての入力パターンに対してテーブルエントリを定義する。未定義のキーは例外を発生させ、サイレントエラーを防ぐ |
| **③ 拡張性（Extensibility）** | 新規申請種別の追加はテーブルへの1行追加のみ。既存ロジックへの影響ゼロ（Open/Closed Principle） |

### BAD vs GOOD 比較

| 観点 | ❌ if-else | ✅ テーブル駆動 |
|------|-----------|--------------|
| 申請種別追加 | 3行のif文追加が必要 | テーブルに1エントリ追加のみ |
| 可読性 | 全18if文を読み解く必要あり | テーブルを見れば全料金が一目瞭然 |
| テスト | 18通りのテストケースが必要 | テーブルデータのテストだけでOK |
| 料金改定 | コードを直接編集 | テーブルの数値を変更のみ |
| バグリスク | コピペミスが発生しやすい | 構造が均一でミスが起きにくい |

## 2. パイプラインとは

### 概念：データが「工程」を順番に通過する

パイプラインとは、**あるデータを複数の処理ステップに順番に通す**設計パターンです。  
各ステップは「前のステップの出力を受け取り、処理して次のステップへ渡す」だけの独立した関数です。

```
入力データ
  │
  ▼
[Step 1: 処理A]  ← 出力1
  │
  ▼
[Step 2: 処理B]  ← 出力2（出力1を入力として受け取る）
  │
  ▼
[Step 3: 処理C]  ← 出力3（出力2を入力として受け取る）
  │
  ▼
最終出力
```

### このシステムで「何がパイプライン」なのか

料金算出エンジンでは、**1つの申請（リクエスト）が11個のステップを順番に通過**します。

```python
# fee_engine.py — パイプラインの実体（抜粋）

def calculate_fee(request: FeeRequest) -> FeeBreakdown:

    # ── Step 1: 基本料金をテーブルから取得 ──────────────────
    base = BASE_FEE_TABLE[(request.app_type, request.applicant_type)]
    # 例: ("F", "法人") → ¥220,000

    # ── Step 2: 地域係数を掛ける ────────────────────────────
    after_region = int(base * REGION_MULTIPLIER[request.region])
    # 例: ¥220,000 × 1.15（政令市） → ¥253,000

    # ── Step 3: 優先度係数を掛ける ──────────────────────────
    after_priority = int(after_region * PRIORITY_MULTIPLIER[request.priority])
    # 例: ¥253,000 × 1.00（通常） → ¥253,000

    # ── Step 4: 複雑度係数を掛ける ──────────────────────────
    after_complexity = int(after_priority * COMPLEXITY_MULTIPLIER[request.complexity])
    # 例: ¥253,000 × 2.10（超複雑） → ¥531,300

    # ── Step 5: 事業規模係数を掛ける（法人のみ）────────────
    if request.applicant_type == "法人":
        after_scale = int(after_complexity * BUSINESS_SCALE_MULTIPLIER[request.scale])
    elif request.applicant_type == "行政":
        after_scale = int(after_complexity * BUSINESS_SCALE_MULTIPLIER["MICRO"])  # 優遇固定
    else:
        after_scale = after_complexity  # 個人はスキップ
    # 例: ¥531,300 × 1.30（中規模法人） → ¥690,690

    # ── Step 6: 季節・繁忙期係数を掛ける ───────────────────
    after_season = int(after_scale * seasonal_multiplier(request.submission_date))
    # 例: ¥690,690 × 1.20（3月） → ¥828,828

    # ── Step 7: 追加料金を加算 ──────────────────────────────
    additional = _calc_additional_fees(request)  # フラグを1つずつチェック
    # 例: 法務¥50,000 + 環境¥100,000 + 書類超過¥125,800 = ¥275,800

    # ── Step 8: 小計 ────────────────────────────────────────
    subtotal = after_season + additional
    # 例: ¥828,828 + ¥275,800 = ¥1,104,628

    # ── Step 9: リピーター割引を引く ────────────────────────
    repeat_discount = int(subtotal * repeat_discount_rate(request.past_applications))
    after_repeat = subtotal - repeat_discount
    # 例: ¥1,104,628 × 0%（初回） → ¥1,104,628

    # ── Step 10: パッケージ割引を引く ───────────────────────
    pkg_discount = int(after_repeat * package_discount_rate(request.sibling_count))
    after_pkg = after_repeat - pkg_discount
    # 例: ¥1,104,628 × 8% → ▲¥88,370 → ¥1,016,258

    # ── Step 11: 100円単位に切り上げて確定 ──────────────────
    final_fee = ((after_pkg + 99) // 100) * 100
    # 例: ¥1,016,258 → ¥1,016,300

    return FeeBreakdown(steps=[base, after_region, ..., final_fee])
```

### なぜパイプライン構造にするのか

| 課題 | パイプラインでの解決策 |
|------|----------------------|
| 係数が8種類あり順序が重要 | ステップの順番がコード構造で自明になる |
| バグがどこで発生したか特定しにくい | 各Stepの入出力を `FeeBreakdown` に記録→ 即座に特定可能 |
| 「季節係数の前か後かで追加料金を足す」が変わりやすい | ステップを差し替え・並び替えるだけで対応できる |
| 個別のステップだけテストしたい | 各Step関数が独立しているため単独でユニットテスト可能 |

### パイプラインの 3 原則（このシステムの場合）

| 原則 | 内容 |
|------|------|
| **① 一方向**（Unidirectional） | 必ず Step1 → Step11 の順に進む。逆流・スキップ（条件付き除く）なし |
| **② 非破壊**（Non-destructive） | 各Stepは元データを書き換えず、新しい値を変数に代入して次に渡す |
| **③ 記録可能**（Auditable） | `FeeBreakdown` オブジェクトが全Stepの中間値を保持。算出根拠を完全に再現できる |

## 3. 段階パイプライン — 11ステップ

```
基本料金 → [Step1] → 地域係数 → [Step2] → 優先度係数 → [Step3]
  → 複雑度係数 → [Step4] → 事業規模係数 → [Step5] → 季節係数 → [Step6]
  → 追加料金加算 → [Step7] → 小計 → [Step8]
  → リピーター割引 → [Step9] → パッケージ割引 → [Step10]
  → 最終確定（100円単位切り上げ） → [Step11]
```

| Step | 処理名 | 使用テーブル / 関数 | 出力例（APP-003）|
|------|--------|-------------------|-----------------|
| 1 | 基本料金取得 | `BASE_FEE_TABLE[(app, user)]` | ¥220,000 |
| 2 | 地域係数適用 | `REGION_MULTIPLIER[region]` | ¥220,000 × 1.15 = ¥253,000 |
| 3 | 優先度係数適用 | `PRIORITY_MULTIPLIER[priority]` | ¥253,000 × 1.00 = ¥253,000 |
| 4 | 複雑度係数適用 | `COMPLEXITY_MULTIPLIER[complexity]` | ¥253,000 × 2.10 = ¥531,300 |
| 5 | 事業規模係数適用 | `BUSINESS_SCALE_MULTIPLIER[scale]` | ¥531,300 × 1.30 = ¥690,690 |
| 6 | 季節・繁忙期係数適用 | `seasonal_multiplier(date)` | ¥690,690 × 1.20 = ¥828,828 |
| 7 | 追加料金加算 | `ADDITIONAL_FEE[key]` 各フラグ独立チェック | +¥275,800 |
| 8 | 小計算出 | 係数後金額 + 追加料金合計 | ¥1,104,628 |
| 9 | リピーター割引控除 | `repeat_discount_rate(past_count)` | ¥0（初回） |
| 10 | パッケージ割引控除 | `package_discount_rate(sibling_count)` | ▲¥88,370（8%） |
| 11 | 最終確定 | `((raw + 99) // 100) * 100` | ¥1,016,300 |

### パイプライン設計の特長

- **順序依存性の明示**: 各Stepは前のStepの出力を入力とする。依存関係がコード構造で自明
- **独立テスト可能**: 各Stepを単独でユニットテスト可能。特定Stepのバグを即座に特定
- **監査証跡（Audit）**: 各Stepの入出力を `FeeBreakdown` オブジェクトに記録。全算出根拠を追跡可能
- **ルール追加が局所的**: 新フラグ追加 = Step7に1条件追加のみ。他Stepに影響なし
- **数値誤差の制御**: 各Stepで `int()` で切り捨て。最終Step11で100円単位に丸め

## 4. 基本料金テーブル

`BASE_FEE_TABLE` — 申請種別 × 申請者区分（18パターン）

| 種別 | 申請種別名 | 個人 | 法人 | 行政機関 |
|------|-----------|------|------|---------|
| A | 建設工事許可申請 | ¥80,000 | ¥150,000 | ¥60,000 |
| B | 営業許可申請（飲食・風俗・古物等） | ¥50,000 | ¥90,000 | ¥40,000 |
| C | 補助金・助成金申請 | ¥30,000 | ¥60,000 | ¥20,000 |
| D | 輸出入許可申請 | ¥70,000 | ¥130,000 | ¥50,000 |
| E | 宅地建物取引業免許申請 | ¥100,000 | ¥180,000 | ¥70,000 |
| F | 産業廃棄物処理業許可申請 | ¥120,000 | ¥220,000 | ¥80,000 |

> **適用ルール**  
> - 新規申請種別（G〜）の追加はこのテーブルに3エントリ追加するだけ  
> - 料金改定はテーブルの数値のみ変更。計算ロジックへの変更は不要  
> - 行政機関は常にMICRO扱い（Step5の事業規模係数が優遇固定）

## 5. 係数テーブル

### REGION_MULTIPLIER — 地域係数（Step 2）

| 区分 | 係数値 | 説明 |
|------|--------|------|
| 過疎地域 `rural` | × **0.85** | 地方過疎地域。手続き促進割引として設定 |
| 都道府県 `prefecture` | × **1.00** | 標準値。全国都道府県窓口 |
| 政令指定都市 `designated_city` | × **1.15** | 窓口対応コスト・物価を反映 |
| 特別区（東京23区） `special_ward` | × **1.30** | 最も手続きが複雑・物価高 |

### PRIORITY_MULTIPLIER — 優先度係数（Step 3）

| 区分 | 係数値 | 標準処理期間 |
|------|--------|------------|
| 通常 `standard` | × **1.00** | 14営業日 |
| 急ぎ `express` | × **1.50** | 7営業日 |
| 最急 `urgent` | × **2.20** | 3営業日（休日作業含む場合あり） |

### COMPLEXITY_MULTIPLIER — 複雑度係数（Step 4）

| 区分 | 係数値 | 書類量の目安 |
|------|--------|------------|
| 簡易 `simple` | × **0.80** | 5枚未満 |
| 中程度 `moderate` | × **1.00** | 5〜20枚（標準） |
| 複雑 `complex` | × **1.45** | 20〜50枚 |
| 超複雑 `extreme` | × **2.10** | 50枚超（法務・環境評価等を伴う） |

### BUSINESS_SCALE_MULTIPLIER — 事業規模係数（Step 5 / 法人のみ）

| 区分 | 係数値 | 規模の目安 |
|------|--------|----------|
| 小規模 `MICRO` | × **0.75** | 資本金300万以下 または 従業員5名以下 |
| 中小企業 `SMALL` | × **1.00** | 資本金300万〜3億・従業員300名以下（標準） |
| 中規模 `MEDIUM` | × **1.30** | 資本金3億〜10億 |
| 大企業 `LARGE` | × **1.65** | 資本金10億超 または 上場企業 |

> ⚠️ 個人申請者はStep5をスキップ。行政機関はMICRO固定（優遇措置）。

## 6. 追加料金テーブル

### ADDITIONAL_FEE — フラット追加料金（Step 7）

| 追加料金項目 | 金額 | 適用条件 |
|------------|------|---------|
| 法務確認料 | ¥50,000 | `legal_review_required = True` |
| 翻訳料 | ¥3,000/枚 | `translation_required = True`。（書類＋添付）枚数 × ¥3,000 |

| 補正対応料（2回目〜） | ¥10,000/回追加 | 例: 3回補正 = ¥15,000 + ¥10,000×2 = ¥35,000 |

## 7. 割引テーブル

### repeat_discount_rate() — リピーター割引（Step 9）

| 過去申請件数 | 割引率 | 適用フェーズ |
|------------|--------|------------|
| 0件（初回） | 0% | — |
| 1〜2件 | **▲5%** | 小計（Step8出力）に対して適用 |
| 3〜5件 | **▲10%** | 小計（Step8出力）に対して適用 |
| 6〜10件 | **▲15%** | 小計（Step8出力）に対して適用 |
| 11件以上 | **▲20%** | 上限割引率 |

### package_discount_rate() — パッケージ割引（Step 10）

| 同時申請件数 | 割引率 | 適用フェーズ |
|------------|--------|------------|
| 1件（単独） | 0% | — |
| 2件同時 | **▲8%** | リピーター割引後（Step9出力）に適用 |
| 3〜4件同時 | **▲15%** | リピーター割引後（Step9出力）に適用 |
| 5件以上同時 | **▲20%** | 上限割引率 |

> ⚠️ **重要**: 2つの割引は**順次適用**（乗算ではない）  
> 例: 小計¥100,000、リピーター10%、パッケージ8%  
> → ¥100,000 × 0.90 = **¥90,000** → ¥90,000 × 0.92 = **¥82,800**
